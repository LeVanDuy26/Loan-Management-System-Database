# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project overview

This is a university thesis project (Khóa luận tốt nghiệp, PTIT, Nhóm 97): an OLTP MySQL 8.0 database that models the full lifecycle of a loan — application intake, credit scoring, multi-level approval, disbursement, installment repayment, and overdue-debt collection — plus a Power BI dashboard for KPI reporting. There is no application backend/frontend; the deliverable is the schema, sample-data tooling, and the dashboard.

The repo is organized into four numbered top-level folders that mirror the thesis structure, not a typical source-code layout:

- `1. DOMAIN KNOWLEDGE/` — business analysis and ontology docs (Markdown/docx) explaining the lending domain the schema models.
- `2. DATABASE DESIGN/` — the actual database: `schema/` (13 ordered `.sql` migrations) and `docs/` (design rationale, ERD, deployment guide).
- `3. SAMPLE DATA/` — standalone Python scripts (no package/CLI) for generating, clearing, monitoring, and profiling data in a running MySQL instance.
- `4. DASHBOARD/` — the Power BI `.pbix` file and a KPI reference doc.

## Commands

Python deps are managed with `uv` (Python >=3.13; see `pyproject.toml` / `uv.lock`):

```bash
uv sync                     # install faker, mysql-connector-python, pandas
# or: pip install faker mysql-connector-python pandas
```

Deploying the schema (migrations must run in numeric order — each file depends on tables created by earlier ones):

```sql
CREATE DATABASE loan_management CHARACTER SET utf8mb4 COLLATE utf8mb4_unicode_ci;
```

```bash
for file in "2. DATABASE DESIGN/schema/"*.sql; do
    mysql -u root -p loan_management < "$file"
done
```

Working with sample data (run from `3. SAMPLE DATA/`, requires schema already deployed):

```bash
python generate_test_data.py   # seeds ~1,000 customers, ~2,000 applications, ~1,400 contracts, plus schedules/collections/collateral
python clear_database.py       # TRUNCATEs all 12 tables (asks for confirmation) — FK checks are disabled/re-enabled around it
python monitor_database.py     # prints a health report: connections, slow queries, table sizes, buffer pool
python profile_database.py     # EXPLAIN + 10x timed benchmark for the key query patterns
```

There is no test suite, linter, or build step in this repo — verification is done by running the migrations against a real MySQL instance and checking `SHOW TABLES` / `SHOW TRIGGERS` / the sample-data scripts.

**Before running any script in `3. SAMPLE DATA/`**, edit the hardcoded `config` dict at the top of that file (`user`, `password`, `host`, `database`) to match your local MySQL instance — there is no `.env` or config file, each script owns its own copy of the connection config.

## Database architecture

12 tables, normalized to 3NF, split into core and supporting entities (see `2. DATABASE DESIGN/docs/database_design.md` for full field-level business rules):

**Core (transactional flow, in order):** `customers` → `loan_applications` → `loan_contracts` → `disbursements` / `repayments` → `collections` (for overdue contracts).

**Supporting (attached to the core flow):** `credit_scores` (customer + application), `approval_workflows` (application, multi-level: Chuyên viên → Trưởng nhóm → Giám đốc), `collaterals` / `guarantors` / `interest_rate_schedules` / `payment_schedules` (all contract-scoped).

Key conventions enforced across the schema:

- `BIGINT` PKs, `DECIMAL(15,2)` for all monetary amounts, `JSON` for flexible fields (e.g., `credit_scores.factors`).
- Business-key numbers (`application_number`, `contract_number`, `disbursement_number`) are auto-generated by `BEFORE INSERT` triggers, not application code.
- CHECK constraints handle static validation (amounts > 0, rates 0–100); anything requiring a lookup in another table or the current date (e.g., "disbursement_date can't be in the future", "scheduled_date can't precede the contract's disbursement_date") is enforced via triggers instead, because MySQL 8.0 CHECK constraints can't reference other tables or non-deterministic functions.
- `013_add_indexes_and_constraints.sql` is the final migration and depends on all 12 tables existing — it adds composite indexes tuned to specific query patterns (e.g., `payment_schedules(due_date, status)` for the overdue-lookup query, `credit_scores(customer_id, score_date DESC)` for latest-score lookup), the cross-table validation triggers described above, a trigger that auto-expires the previous active `interest_rate_schedules` row when a new active one is inserted, a trigger that flips `loan_contracts.status` to `closed`/`defaulted` based on `payment_schedules` state, and the `vw_loan_summary` reporting view (joins contracts → applications → customers → disbursements/repayments/payment_schedules).
- When adding a new migration, append it as `014_...sql` and keep numeric ordering — files are applied sequentially and later ones assume earlier schema exists.
